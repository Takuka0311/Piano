# Spec-Driven 开发说明

## 目的
- 所有新增/变更功能先写规格，再进入实现。
- 用规格保证旧版兼容目标和重构质量可追踪。

## 规格文档模板
- 标识：`spec-xxx-<topic>.md`
- 必填章节：
  - 背景与目标
  - 范围（In/Out）
  - 术语与数据定义
  - 功能需求（可验证）
  - 非功能需求（性能/稳定性）
  - 验收标准
  - 测试计划
  - 风险与回滚

## 状态流转
- `draft`：草案，需求讨论中。
- `review`：进入评审，冻结主要目标。
- `approved`：可实施。
- `implemented`：代码完成并验证通过。
- `archived`：被替代或废弃。

## 变更规则
- 任何影响兼容行为的改动，必须先改对应 spec。
- 已 `approved` 的 spec 若变更，需记录“变更原因/影响范围”。
- 代码 PR 需引用对应 spec 文件路径。

## 追踪建议
- 在 PR 描述中添加：
  - `Spec`: 关联 spec 路径
  - `Status`: 当前阶段
  - `Validation`: 执行的验收项
