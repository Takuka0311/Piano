# 重构技术选型

## 选型目标
- 仅 Windows 平台。
- 依赖尽量少，构建和运行尽量简单。
- 执行效率和音频稳定性优先。
- 与旧版行为尽量全量兼容（键位、曲谱、速度/调式、VSTi 路径）。

## 参考实现提炼（`freepiano-*`）
- 参考目录显示该系列长期采用 C/C++ 原生栈，具备：
  - 多音频后端（DirectSound/WASAPI/ASIO）。
  - MIDI 输入输出。
  - VSTi 接入。
  - 媒体导出（MP4/WAV，含 x264/mp4v2/faac 依赖）。
- 经验结论：
  - “实时演奏核心”与“导出能力”可拆层，避免重依赖污染主链路。
  - Windows 音频后端应优先 WASAPI，DirectSound 作兼容。

## 候选路线对比

| 路线 | 依赖体量 | 开发效率 | 性能/低延迟 | 与目标匹配 |
|---|---|---|---|---|
| A. 轻量图形库 + WASAPI（可选 DS）+ 自建核心 | 小 | 中 | 高 | 高 |
| B. Qt Widgets/Qt6 + WASAPI（可选） | 中 | 高 | 中高 | 中 |
| C. JUCE 全家桶 | 中大 | 中 | 高 | 中低 |

## 推荐路线（最终）
- **采用路线 A**：轻量图形库 + WASAPI 主后端 + DirectSound 兼容层 + VSTi 适配层 + 可选 MIDI In/Out。
- 说明：
  - 你已明确希望轻量 GUI 且减少 VS 绑定，本路线最符合目标。
  - 音频主链可保持高性能与低抖动。
  - 通过分层设计保留后续扩展空间（例如再接 Qt 或 ASIO）。

## 关键技术决策
- GUI：
  - 选择轻量图形库实现主界面，聚焦钢琴键盘可视化与配置面板。
  - 不引入 Qt 运行时作为新版本硬依赖。
- 音频：
  - 默认 WASAPI（共享模式起步，保留低延迟参数）；DS 作为回退。
- MIDI：
  - 首期保留 MIDI Out，MIDI In 作为 P1（但接口首版即预留）。
- 乐器：
  - 首期保留 VSTi 接入能力；旧版 `mdaPiano.dll` 作为兼容样本。
- 导出：
  - MP4 导出放到后续版本，先实现 WAV 导出与可测试离线渲染。

## 不选项与理由
- **不选 Qt 作为新主路线**
  - 违背“依赖尽量少、运行尽量简单”的核心目标。
- **不选 JUCE 作为首发主栈**
  - 对当前需求偏重，工具链与工程复杂度上升明显。
- **不将 ASIO 作为首发必选**
  - 驱动/设备差异大，首发维护成本高；可后续作为增强。
