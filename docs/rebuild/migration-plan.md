# 重构迁移计划

## 迁移原则
- 先兼容、再优化、后扩展。
- 每个阶段都可独立验收和回滚。
- 关键行为以旧版曲谱和键位数据为基线。

## 阶段划分

### Phase 0：需求冻结与基线采样
- 冻结旧版兼容目标（功能与语义）。
- 建立基线样本：
  - `init.keyboard`
  - `カワキヲアメク.keyboard`
  - `カワキヲアメク.in`
  - `梦中的婚礼.in`
- 输出行为对照表（输入 -> 预期输出）。

### Phase 1：核心引擎落地（无复杂 UI）
- 完成 `*.keyboard`/`*.in` 解析器与事件调度器。
- 完成命令总线与引擎状态机。
- 接入至少一种音频输出（WASAPI）和一种乐器路径（VSTi 或 MIDI out）。
- 验收：命令行/最小窗口下可回放基线曲谱。

### Phase 2：轻量 GUI 与交互闭环
- 完成键盘可视化、按键映射编辑、播放控制。
- 对齐旧版核心交互（按压反馈、布局切换、停止恢复）。
- 验收：手动演奏与自动播放均可稳定运行。

### Phase 3：兼容增强与工程化
- 加入 DirectSound 回退路径。
- 补齐配置持久化、错误提示、日志。
- 建立 CI 发布链，输出可运行 exe 包。

### Phase 4：增强能力（可选）
- MIDI In、WAV/MP4 导出、更多脚本命令。
- 性能优化与可观测性增强。

## 风险与应对
- 解析兼容风险：先做“旧行为测试集”，每次改动跑回归。
- 音频稳定性风险：保持音频线程最小逻辑，所有重操作移出回调。
- 插件兼容风险：VSTi 适配层单独隔离，明确错误恢复路径。

## 里程碑验收标准
- M1：可解析并正确调度旧版曲谱。
- M2：可听见正确音高/节奏，支持基础交互。
- M3：CI 可产出可运行 exe。
- M4：文档与规格齐全，可进入持续 spec-driven 迭代。
